# PicoClaw 角色扮演模块实现方案

## 概述

为路由器上的 picoclaw (Go AI 助手) 添加了完整的角色扮演支持，兼容 SillyTavern/酒馆角色卡格式（V1/V2/V3）。用户可导入 PNG 角色卡、通过自然语言管理角色库、随时切换角色身份。

## 角色卡数据流

### 导入 PNG 角色卡
```
用户拖入 PNG → FileReader.arrayBuffer() → parseCharacterCardPNG()
  → 遍历 PNG chunks → 找 tEXt 块 (keyword=chara/ccv3)
  → Base64 解码 → JSON.parse → normalizeCard()
  → 存入 localStorage → renderCharList()
  → 用户点击激活 → cardToPrompt() → POST /character/clear?name=XXX
  → Go 写入 SOUL.md + characters/{id}.md + _registry.json
```

### 聊天框导入角色
```
用户发送格式化文本 → Agent 解析 Key("value") 格式
  → write_file characters/{id}.md
  → write_file _registry.json
  → write_file SOUL.md → 立即切换
```

### 自然语言切换
```
用户说 "切换到 HuTao" → Agent:
  1. read_file characters/_registry.json
  2. read_file characters/hutao---genshin-impact.md
  3. write_file SOUL.md (写入角色提示词)
  4. write_file _registry.json (更新 active_char)
```

## 提示词注入机制

picoclaw 使用五层 Prompt 架构，角色内容注入在最高层：

```
Layer:  kernel    (优先级 100)   ← "你是 picoclaw" 身份
Layer:  kernel    (优先级 100)   ← characterPrompt（角色卡内容，前置到身份之前）
Layer:  instruction (优先级 80)  ← SOUL.md + AGENT.md（引导文件）
Layer:  capability (优先级 60)   ← 工具/技能
Layer:  context   (优先级 40)    ← 记忆/动态上下文
```

当角色激活时，SOUL.md 的内容以 `CRITICAL: You are now ROLEPLAYING...` 开头，强制 LLM 以角色身份回应。

Prompt 缓存通过 `cachedCharacterPrompt` 字段对比自动失效，SOUL.md mtime 变化也会触发重建。

## 角色注册表格式

`workspace/characters/_registry.json`:
```json
{
  "characters": [
    {"id": "xing-qiu", "name": "Xing Qiu", "preview": "Xing Qiu"},
    {"id": "hutao---genshin-impact", "name": "HuTao | Genshin Impact", "preview": "HuTao | Genshin Impact"}
  ],
  "active_char": "hutao---genshin-impact"
}
```

## Agent 可执行操作

通过在 `AGENT.md` 中注入角色系统说明，Agent 可以自主完成：

| 用户指令 | Agent 工具调用 |
|---------|--------------|
| "角色卡列表" | `read_file characters/_registry.json` |
| "切换到 Xing Qiu" | `read_file` → `write_file SOUL.md` → `write_file _registry.json` |
| "恢复默认" | `write_file SOUL.md` (默认 PicoClaw) |
| "Xing Qiu 背景" | `read_file characters/xing-qiu.md` |
| 发送 Name("Venti")... | 解析 → `write_file` → 注册 → 激活 |
| "删除 HuTao" | 删 `characters/{id}.md` → 更新 registry |

## 编译与部署

```powershell
# Windows 交叉编译 MIPS 小端序
cd picoclaw
$env:GOOS='linux'; $env:GOARCH='mipsle'; $env:GOMIPS='softfloat'; $env:CGO_ENABLED='0'
go build -ldflags="-s -w" -o picoclaw_final.exe ./cmd/picoclaw/

# 部署（Z: 盘映射路由器 U 盘）
Copy-Item picoclaw_final.exe Z:\picoclaw_final.exe
# SSH 停止 → cp 到 /mnt/sda1/picoclaw_new → 启动
```

WebUI 热更新直接写 Z 盘，路由端 cp 到 `/tmp/picoclaw_webui.html` 即可生效，无需重启。

## 踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| LuCI 模板报错 "unfinished string" | PowerShell `Out-File -Encoding UTF8` 加了 BOM | 用 `UTF8Encoding($false)` 无 BOM 写入 Z 盘 |
| ucodebridge 解析器拒绝大改 | 对 `<% %>` 块结构变化敏感 | 只做最小增量修改，保持原始模板骨架 |
| character_prompt JSON 字段未生效 | Go 配置加载路径复杂 | 改用 SOUL.md 原生引导文件机制 |
| `async async` 双关键字 | replace_in_file 匹配到了相邻块的 async | 发现后用 sed 直接修复，不再用 replace_in_file 拼接复杂 JS |
| bridge.lua 被 PowerShell 损坏 | 字符串拼接时引号/转义被改动 | 纯 Lua 文件只能通过 write_to_file 写，不能用 PowerShell 文本处理 |
