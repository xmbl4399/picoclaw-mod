# PicoClaw 角色扮演模块方案

## 架构概述

PicoClaw Mod 在原生 PicoClaw 上增加了 Galgame 风格的角色扮演能力，核心为 **"角色切换 → 会话隔离 → 角色记忆"** 三层架构。

## 模块详情

### 1️⃣ 角色管理子系统 (health server)

**入口**: `pkg/health/server.go`

```
GET  /character/active  → { "character_id": "kirino" }
POST /character/active  → { "character_id": "kirino" } (需 Bearer token)
GET  /character/model   → 当前模型配置
```

**核心数据结构**:
```go
type Server struct {
	activeCharacter        atomic.Value   // string; "" = no character
	characterChangeCallback func(string)  // 角色切换时回调
	agentLoop              *AgentLoop
}
```

**回调链**:
```
POST /character/active
  → server.SetActiveCharacter(id)
    → characterChangeCallback(id)
      → (在 gateway.command.go 中注册)
        → for each channel: channel.SetCharacterGetter(server.CharacterGetter())
        → for each AgentInstance:
            instance.SetCharacterID(id)
            → contextBuilder.WithCharacterID(id)
            → memoryStore.DisconnectCharacterID(id)  (切换 memory 子目录)
        → UpdateAgentConfig()  (如果需要)
        → BroadcastCharacterSwitch(id)
```

### 2️⃣ 会话隔离子系统 (session)

**数据流**:

```
用户消息 → QQ/Pico/Telegram通道
  → channels.BaseChannel.HandleMessageWithContext()
    → c.characterGetter() 获取当前角色ID
    → inboundCtx.Raw["character_id"] = "kirino"
  → bus.PublishInbound(msg)
  → session/allocator.go: buildSessionScope()
    → dimensions = ["chat", "character"]
    → dimensionValues["chat"] = "pico:xxx"
    → dimensionValues["character"] = "kirino"
  → session/key.go: BuildSessionKey(scope)
    → 过滤空维度值
    → "chat=pico:xxx&character=kirino" → HMAC → sessionKey
```

**关键规则**:
- 空 `character_id` → 该维度被跳过 → 旧消息和角色的老的 sessionKey 不变
- 角色切换后，新消息自动带 `character_id` → 新的 sessionKey
- 每个角色的聊天历史完全隔离

### 3️⃣ Memory 隔离子系统

**路径规则**:

| 角色模式 | Memory 路径 |
|---------|------------|
| 无角色 (characterID="") | `workspace/memory/MEMORY.md` |
| 角色 kirino | `workspace/memory/characters/kirino/MEMORY.md` |

### 4️⃣ RULES.md — 永久规则

独立于 SOUL.md 的固定规则文件，**切换角色时保留不变**。存储在 `workspace/RULES.md`。

### 5️⃣ Prompt 结构（ContextBuilder）

```
System Prompt:
├── 静态前缀（缓存命中区）: IDENTITY → SOUL → RULES → USER → MEMORY
├── 动态后缀（每请求变化）: 时间 / 会话 / 发送者信息
└── 对话历史（按轮次追加）
```

**缓存跟踪**: `sourcePaths()` 跟踪 SOUL.md + RULES.md + MEMORY.md 的 mtime。

## 新增特性（v2026-07-21）

### 🎙️ QQ TTS 语音输出

角色台词自动合成为 QQ 语音气泡发送。

**工作流**:
```
角色发台词 → QQ.Send() 完成文本发送
  └→ sendTTSAsync() (异步 goroutine)
       ├→ 正则提取台词 (@说/角色名：格式)
       ├→ 过滤选项行 (@选1/@选2/@选3)
       ├→ Mimo TTS → mp3 → MediaStore
       └→ SendMedia() → MP3帧头解析时长 → file_type=3 语音气泡
```

**关键文件**:
| 文件 | 功能 |
|------|------|
| `pkg/channels/qq/qq.go` | Send() 末尾异步调用 + sendTTSAsync() |
| `pkg/channels/qq/init.go` | tts.DetectTTS(cfg) 注入 TTS provider |
| `pkg/channels/qq/audio_duration.go` | MP3 ID3v2 标签跳过 + MPEG 帧头解析 → 毫秒时长 |
| `pkg/channels/pico/pico.go` | sendTTSAsync() 同样过滤 @选 选项行 |

### ⚡ Token 缓存优化

参考 DeepSeek 前缀匹配缓存原理，提升对话缓存命中率。

**核心改动**:

| 改动 | 文件 | 效果 |
|------|------|------|
| 滑动窗口压缩 | `context_legacy.go` | forceCompression 从丢 50% → 丢 1 个最旧 turn |
| 缓存命中监控 | `pipeline_llm.go` | 日志输出 cache_hit_tokens / cache_hit_ratio |
| Usage 扩展 | `protocoltypes/types.go` | 新增 PromptCacheHitTokens / PromptCacheMissTokens |

**压缩策略对比**:
```
旧: [sys | t1 | t2 | t3 | t4 | t5] → 50% → [t3 | t4 | t5]  前缀完全断裂 ❌
新: [sys | t1 | t2 | t3 | t4 | t5] → 滑窗 → [t2 | t3 | t4 | t5]  仅丢 t1 ✅
```

### 🔧 CGI 角色管理端点

## 客户端实现

### Galgame WebUI

- 单 HTML 文件 (~36KB)
- 零框架，Vanilla JS + CSS
- WebSocket 通信 (`new WebSocket(url, 'token.picoclaw-router-2026')`)
- localStorage 按角色隔离聊天历史
- 内嵌 PNG 角色卡导入器 (SillyTavern V2/V3)，支持 `<!-- char_id:xxx -->` 自标记
- GAL 格式渲染（@场景/人物/说/心/旁白/回忆/提示/选1-3）
- Markdown 渲染 + Token 计数器 + 流式文本输出

### 角色切换流程

```
用户点击角色 pill 按钮
  → POST /character/active (Bearer token)
  → 写入 SOUL.md 带 <!-- char_id:id --> marker
  → 持久化 _registry.json (active_char)
  → 服务端回调链更新 session 维度
  → 切换 localStorage 历史 key
  → 发送消息 → 新的 session key → 新聊天上下文
```

### 角色读取优先级

```
SOUL.md marker (权威) → registry active_char (fallback) → '' (默认)
```

## 已知问题与教训

### Pico Token 认证
- WebUI 的 TOKEN 必须是原始字符串 `picoclaw-router-2026`，不是 base64
- 服务端 `matchedSubprotocol` 做的是字符串相等比较

### BusyBox 限制
- `sed` 的 `\b` 被解释为退格符，而非 word boundary
- 没有 `python3`；已配置 `custom_deny_patterns: ["python3"]`
- 没有 `jq`

### OverlayFS 陷阱
- 配置文件存在两个物理位置：`/root/.picoclaw/config.json`（overlay upper）和 `/overlay/upper/root/.picoclaw/config.json`
- 直接通过 SSH 写 `/root/.picoclaw/config.json` 即可（路径自动解析到 overlay upper）

### CGI 权限
- `uhttpd` 以 `nobody` 运行，shell 解释器需要**读**权限才能执行脚本
- 权限必须是 755（r-x for other），**不能是** 711（x only）
